# Migration from v2

A guide for projects already using the script-tag SDK (`wall.x.x.js` via `<script>` + iframe rendering) that want to move to SDK 3.0 (bundled npm package + Shadow DOM, no iframe).

<Callout type="info">
  **You don't have to migrate right now.** The v2 SDK is still supported and keeps working. Migrate when:
  - **You're building a Chrome extension (MV3)** — CWS review forbids remote code, the iframe SDK won't pass. SDK 3.0 is the only path.
  - **You want to customise the UI** or fit the paywall into your own layout — Shadow DOM is friendlier than an iframe.
  - **Bundle size matters** (Telegram Mini Apps, edge runtimes) — 3.0 ships via npm and tree-shakes.
  - **You want a typed API** — 3.0 is fully TypeScript with strict event payloads and discriminated unions.
</Callout>

  **`apiOrigin` is required at init.** v2 baked the origin into the script; v3 makes you pass your `custom_domain` explicitly. Without it, `BillingClient` throws `invalid_config`. Set up a [custom domain](/docs-v2/custom-domains) first if you haven't.
</Callout>

## Migration strategy

You don't have to migrate everything at once:

1. **In-place upgrade.** Flip a single paywall's `sdk_version` in the DB; the online server is channel-agnostic. After release, verify new sessions go through SDK 3.0.
2. **Parallel paywalls.** Create a second paywall with the same prices and route SDK 3.0 to part of the traffic (A/B by feature flag).
3. **Full cutover.** Ship a frontend with SDK 3.0, strip the old `<script>` tag and iframe assets.

The optimal path is usually **in-place upgrade** on one paywall; the legacy v2 SDK keeps serving users already signed in from previous sessions through the same endpoints — no data migration needed.

### Step 6. Replace `<script>` tags with imports

Remove `<script src="...appbox.space/...">` from your HTML. If you have several call sites, centralize the init in one module:

```ts
// src/paywall.ts
import { PaywallUI } from '@monetize.software/sdk';

export const paywall = new PaywallUI({
  paywallId: process.env.NEXT_PUBLIC_PAYWALL_ID!,
  apiOrigin: process.env.NEXT_PUBLIC_PAYWALL_ORIGIN!,
  auth: true
});
```

Import `paywall` from this module wherever needed. `PaywallUI` is singleton-friendly.

## Auth: what changes

v2 ran auth inside an iframe on appbox.space — tokens lived in cookies of that origin and your code couldn't see them. v3 uses Bearer tokens in `AuthClient` stored in `localStorage` (web) / `chrome.storage.local` (extension). That gives you:

- ✅ Working Chrome extensions and Telegram Mini Apps (cookies are either forbidden or partitioned there).
- ✅ Direct access to `access_token` for requests to **your** backend (`Authorization: Bearer …`).
- ✅ Typed events `INITIAL_SESSION / SIGNED_IN / SIGNED_OUT / TOKEN_REFRESHED / USER_UPDATED / PASSWORD_RECOVERY`.
- ⚠️ Sensitivity to host-page XSS — tokens are reachable to JS. See [Security](/docs-v2/sdk-v3/security#token-storage-and-xss) for mitigations.

### What to do

1. If you had your own backend auth — in v3 you can stay in **hybrid identity** mode (`identity: { email, userId }` without managed-auth) or switch to our `AuthClient` (`auth: true`). Hybrid is convenient when the user is already signed in via your app.
2. If you used `paywall.open({ resolveEvent: 'signed-in' })` for the login flow — it's now `paywall.openSignin()` + `paywall.on('authChange', cb)`.
3. If you kept `appbox.space` cookies somewhere for server-side recovery — no longer needed. Bearer and the server-side `apiKey` (on `BillingClient`) fully replace it.

## Chrome Extension

v2 does **not work** in Chrome MV3 — CWS review forbids remote code execution and `wall.x.x.js` is loaded from appbox.space on the fly. The only path is SDK 3.0 + the dedicated `@monetize.software/sdk-extension` package with an offscreen document.

See the dedicated guide: [Quickstart → Chrome Extension](/docs-v2/sdk-v3/quickstart#chrome-extension-mv3) and [Installation → Chrome Extension](/docs-v2/sdk-v3/installation#chrome-extension-mv3).

## CSP

v2 required allowing an iframe on `appbox.space`. v3 doesn't need that, but your `apiOrigin` (custom_domain) shows up — add it to `connect-src`:

<Tabs items={['v2', 'v3']}>
<Tabs.Tab>

```http
Content-Security-Policy:
  script-src 'self' https://appbox.space;
  frame-src https://appbox.space;
  connect-src https://appbox.space;
```

</Tabs.Tab>
<Tabs.Tab>

```http
Content-Security-Policy:
  script-src 'self';
  connect-src 'self' https://pay.your-domain.com;
  img-src 'self' https://pay.your-domain.com data:;
  style-src 'self' 'unsafe-inline';
```

`'unsafe-inline'` in `style-src` is required because Shadow DOM SDK injects its Tailwind bundle as a `<style>` tag. CSP nonces are not supported yet — known limitation. If your security policy strictly forbids `unsafe-inline`, stay on v2 for now.

</Tabs.Tab>
</Tabs>

## Can both SDKs run side by side?

**On the same paywall** — no: one paywall = one channel (`sdk_version` is stored in the DB; the online server serializes bootstrap for the right format). Flipping between channels is fine, but at any moment the paywall is either v2 or v3.

**On two different paywalls (e.g. staging vs prod)** — yes, easy. The same `paywallId` cannot be served to both SDKs simultaneously; different `paywallId`s are fine.

## Post-migration checklist

- [ ] All `<script src="...appbox.space/...">` tags removed from HTML.
- [ ] `apiOrigin` points at your [custom domain](/docs-v2/custom-domains).
- [ ] `user.paid` → `user.has_active_subscription` everywhere (grep the codebase).
- [ ] `window.addEventListener('message', ...)` handlers replaced with `paywall.on(...)`.
- [ ] If you used `setUserMetadata`, `userMeta` is passed into auth methods.
- [ ] CSP: `connect-src` includes `apiOrigin`, `frame-src appbox.space` is removed.
- [ ] [Security → Production checklist](/docs-v2/sdk-v3/security#production-checklist) is green.
- [ ] Webhook handler still works (payload format does not depend on the SDK channel).
- [ ] `paywall.open()` → `purchase_completed` → unlock tested on the test payment processor.
